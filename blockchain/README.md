# Smart Contracts

The `blockchain/` directory contains the Solidity smart contracts and build utilities that form the decentralized authorization and consensus backbone of the PRISM architecture.

---

## Directory Structure

```text
blockchain/
├── build/                      # Compiled ABI and bytecode (JSON) generated by Solc
└── contracts/                  # Solidity source code
    ├── Registry.sol            # Decentralized identity, enrollment, and access control
    ├── Permissions.sol         # QBFT connection authorization and node management
    ├── Attendance.sol          # Decentralized attendance logging
    ├── EnumerableSet.sol       # Library: Iterable set utilities for authorized nodes
    └── build.py                # Python wrapper for compiling contracts via Py-Solc
```

---

## Contracts

PRISM relies on interacting smart contracts to securely manage identities and network participation.

### 1. `Registry.sol`
Serves as the central repository for Decentralized Identifiers (DIDs) and network roles. It handles:
- **Bijective Device-to-Node Mapping**: Ensuring each biometric kiosk maps to exactly one consensus enode (`registerKioskEnode`).
- **Access Control Authorities**: Granting node-specific permissions like `canEnroll` (registering users) and `canRecord` (submitting biometric attendance logs).
- **User Lifecycle Management**: Enrolling, suspending, and revoking user credentials securely.
- **Consent-Based Reissuance**: Requiring cryptographic vouchers (`REISSUE_CONSENT`) signed by the user's private key to authorize credential re-generation.

### 2. `Permissions.sol`
Operates directly with Quorum's QBFT consensus engine to ensure strict network boundaries:
- **Node Permissioning**: Explicitly allows or denies nodes from participating in block validation via `addNode` and `banNode`.
- **Connection Authorization**: Provides the `connectionAllowed` callback for the Quorum node to drop peer-to-peer TCP connections with unauthorized or banned nodes. It isolates banned managers by querying the `Registry` cross-contract.

### 3. `Attendance.sol`
A lightweight logging contract that acts as the entry point for kiosks submitting attendance events:
- **Verification**: Validates directly with the `Registry` that the submitting kiosk is authorized (`canRecord`), and the target User is active and not suspended.
- **Immutability**: Emits the `Logged` event containing the user's DID, timestamp, and IPFS `infoCID` containing the biometric match data.

### 4. `EnumerableSet.sol`
A standard library utility (adapted from OpenZeppelin) utilized by `Permissions.sol`. 
- **O(1) Efficiency**: It provides O(1) tracking, insertion, and removal logic for the `_authorizedNodes` set. Unlike standard Solidity arrays, it allows efficient lookup and deletion without iterative loops, minimizing gas consumption for node administration.

### 5. Build Automation: `build.py`
A custom Python automation script that handles compilation organically using `py-solc-x`. 
- **EVM Artifact Extraction**: It compiles the `.sol` files, extracts the ABI and deployable bytecode, and writes them out as cleanly formatted JSON artifacts into the `build/` directory for consumption by the PRISM backend orchestrator.

## Smart Contract Guard & Threat Mitigation

PRISM enforces highly deterministic security protocols. The system defends against various exploitation attempts logically via contract reverts.

### Table 1: Attack Vector → Guard Mitigation

| # | Attack Vector | Mitigation | Layer |
|:---|:---|:---|:---|
| 1 | Unauthorized enode registration | Device-to-node mapping requires prior consensus-layer authorization before registry linkage | Registry |
| 2 | Non-manager role impersonation | Role-based access control rejects callers without explicit manager privileges | Consensus |
| 3 | Manager delegation abuse | Manager appointment restricted to contract owner with explicit grant/revoke semantics | Registry |
| 4 | Owner self-ban | Ban operation reverts when target is the contract owner | Registry |
| 5 | Administrative zombie (banned manager persists) | Cross-contract isolation — consensus layer queries registry ban status before authorizing operations | Cross-contract |
| 6 | Pending owner griefing | Ban operation reverts when target is a pending ownership transfer recipient | Registry |
| 7 | Mapping overwrite (1:1 constraint violation) | Bijective mapping enforced — each device maps to exactly one node and vice versa | Registry |
| 8 | Unlinked device bypass (procedural fail-open) | Authority grants require prior device-to-node linkage; unregistered devices cannot receive permissions | Registry |
| 9 | Unauthorized credential reissuance | Reissuance restricted to callers with enrollment authority | Registry |
| 10 | Outbound dialing to banned node (gossip leak) | Connection authorization check rejects outbound connections to banned nodes | Consensus |
| 11 | Suspension bypass / duplicate suspension | Suspension checks active enrollment status; duplicate suspension reverts | Registry |
| 12 | Consent-less credential reissuance | Reissuance requires a cryptographic consent voucher signed by the credential holder | Registry |
| 13 | Forged consent signature | ECDSA signature recovery validates signer identity; mismatched signer causes revert | Registry |
| 14 | Voucher replay (consumed nonce) | Per-user nonce increments atomically on each reissuance; stale nonces are rejected | Registry |
| 15 | Suspended user reissuance | Reissuance checks suspension status before processing | Registry |
| 16 | Premature revocation finalization | Finalization enforces a configurable grace period via block-timestamp check | Registry |
| 17 | Reissuance during pending revocation | Reissuance checks for active revocation request before processing | Registry |
| 18 | Nonce double-spend (same voucher, two transactions) | Nonce consumed on first use; identical voucher submitted in a second transaction reverts | Registry |
| 19 | Manager deadlock (owner loses administrative access) | Owner retains fallback manager privileges regardless of explicit manager state | Registry |
| 20 | Cross-contract state desync | Atomic revocation function clears device mapping and strips all associated permissions in a single transaction | Cross-contract |
| 21 | Signature malleability (s-value canonicalization) | Signature recovery rejects non-canonical s-values (s > secp256k1n/2), enforcing EIP-2 compliance | Registry |
| 22 | Malformed payload bloat (gas limit attack) | Oversized enrollment payload exceeds block gas limit, causing transaction revert | Registry |

## Build and Deployment Workflow

The contracts are meant to be compiled and deployed organically as part of the overall cluster orchestration logic.

To deploy these contracts to the live PRISM Quorum network:
```bash
python provision/cluster_manager.py deploy
```
This single command triggers the internal `build.py` compilation, automatically links the backend deployment wallet, broadcasts the state, and synchronizes the deployed addresses back to `master.yaml` for node ingestion.
